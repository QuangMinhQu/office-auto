# Agent Decomposition & Orchestration

Modern agent frameworks advocate **breaking tasks into subtasks** handled by specialized agents. Research shows that large LLM tasks benefit from decomposition: instead of one monolithic prompt, use a workflow of agents coordinating steps【43†L37-L46】. For example:

- **Chain-of-Agents (CoA):** Google’s Chain-of-Agents breaks a long input into chunks and assigns *worker agents* to each, passing intermediate results along a chain, then a *manager agent* synthesizes the final output【39†L201-L210】【39†L239-L249】. This sequential multi-agent approach reduces context blow-up (time complexity goes from *n²* to *n·k* for input length *n* and LLM window *k*【39†L207-L211】).
- **ACONIC decomposition:** Recent work models tasks as constraint satisfaction and uses tree decomposition to split them into minimal consistent subproblems【43†L53-L62】. This ensures each subtask is easier, yet globally consistent.  
- **OpenCode/Office pipelines:** Practical tools like OpenCode (an open-source coding agent) include built-in primary and subagents for specific roles【21†L223-L231】【21†L233-L239】. For instance, OpenCode’s “general” subagent handles multi-step research in parallel, while “explore” is a fast read-only search agent, and “compaction” compresses long context automatically【21†L223-L231】【21†L233-L239】. 

**Applied to MD→DOCX:** We should design a multi-agent workflow. For example: a *Planner* agent first analyzes the Markdown and decides how to map sections to the template; *Parser* agents extract structured data from the Markdown (headings, lists, etc.); a *Formatter* or *Writer* agent then uses a document tool (e.g. OfficeCLI) to fill the template. Finally a *Validator* agent checks the output (using `officecli validate` or screenshots). This mirrors team-based architectures (“Leader coordinates planning, coding, review” as in Open Office)【35†L63-L72】.

# Context & Token Optimization

The current workflow uses ~130k tokens in one session, which is unsustainable. To reduce token usage, we can leverage **external tools and grounded function interfaces (GFIs)** so that heavy content isn’t fed into the LLM context:

- **OfficeCLI with MCP server:** OfficeCLI offers a built-in Model Context Protocol (MCP) server. Starting `officecli mcp ...` exposes all document operations as JSON-RPC tools【51†L698-L706】. An LLM agent with MCP support can call these tools (like `get`, `set`, `merge`) without the document text ever entering the prompt. This avoids tokenizing the entire DOCX content.  
- **Template merge / batch operations:** Instead of editing the doc step by step in context, use OfficeCLI’s `merge` command to fill placeholders in a template. As OfficeCLI docs note: `merge` replaces `{{key}}` fields with JSON data *deterministically and cheaply (zero token cost)*【49†L614-L623】. For example, convert Markdown to an intermediate JSON and then run `officecli merge report-template.docx report.docx data.json`. This “fill once/many times” strategy minimizes LLM involvement. Similarly, OfficeCLI’s `batch` or resident mode can apply many edits in one command, further reducing prompts【47†L893-L902】【49†L642-L651】.
- **Summarization & selective retrieval:** Pre-summarize or chunk the Markdown before sending to the agent. For instance, use an MCP tool that reads the MD and returns a concise summary (e.g. first 200 tokens) rather than full text【64†L329-L337】. Only if more detail is needed does the agent call a `get_full_md_content()` tool. This avoids loading all content. Also maintain minimal context by pruning old information (MindStudio notes “selective context injection” to drop irrelevant data and keep summaries【64†L369-L378】).  
- **Batching calls:** Combine related edits into one tool call when possible. E.g., rather than separate `set` commands for each paragraph, use OfficeCLI’s JSON `batch` input to group edits【47†L893-L902】. This cuts the overhead of multiple tool invocations【64†L344-L353】.

Other tools: Traditional converters can operate outside the LLM (no tokens). For example, **Pandoc or Quarto** can convert Markdown to DOCX via templates without an LLM step (the agent could simply shell out or call these). Using such CLI tools means the AI only needs to invoke them and verify output, not process all text. The Reddit discussion even suggests Pandoc for MD→DOCX conversion as a preprocess【55†L215-L223】 (though we won’t cite Reddit directly, it’s a known fact that Pandoc/Quarto handle format conversion).

# Proposed Subagent Workflow

Based on these ideas, a scaled-down pipeline might look like:

1. **Input Agent (Pre-processor):** Reads the user’s MD file (via an MCP document reader or code read). It segments the content (by section or size) and optionally summarizes it. This agent tags content types (headings, paragraphs, tables). It writes summaries to memory.  
2. **Planner Agent:** Examines the extracted outline or JSON structure of the MD. It determines how each part should map into the DOCX template. For example, “the first heading fills the title, the second is a section header,” etc. It may use Chain-of-Agents style: splitting the MD into logical chunks (intro, methods, results, etc.) and routing each to different worker agents if needed【39†L215-L223】【39†L254-L262】.  
3. **Document Builder Agents:** One or more specialized subagents execute the actual document edits. For instance, an “Office Writer” subagent using the OfficeCLI MCP tools: it receives a JSON plan (e.g. “add paragraph with text X at path /body”) and calls `officecli add`/`set` over MCP. Because the tool calls use JSON commands, the LLM doesn’t need to see raw text. We could have parallel agents if the doc is large (each working on different sections) with separate OfficeCLI instances or contexts.  
4. **Formatter/Validator Agent:** After assembly, this agent validates formatting and consistency. It might call `officecli validate report.docx` and read issues【47†L913-L922】, or even `officecli view report.docx screenshot` to get a snapshot (which the agent can analyze with vision tools). If errors occur (e.g. style mismatch), it instructs the Writer agent to fix them.  
5. **Finalizer Agent:** If templates used `{{key}}` placeholders, a final agent could fill any remaining fields (e.g. dynamic tables or charts) via `officecli merge`. Or it could run Pandoc to produce final formatting touches.  

Throughout, intermediate results (like summaries or partial JSON) are stored in the agent’s memory or a workspace so that each step’s context stays small. We also ensure to **disable sending large file contents** directly in prompts – the MCP/tools handle reading/writing.

# Integration Plan & Metrics

To implement this, one would:

- **Configure OfficeCLI MCP:** Ensure `officecli mcp` is running (for the LLM’s backend, e.g. Qwen) and the agent has the OfficeCLI SKILL.md loaded. This lets the model call commands like a function, with no context cost【51†L698-L706】.  
- **Agent Prompts & Skills:** Write concise agent prompts or SKILL.md files that instruct each subagent’s role. Follow patterns from anthropic’s skills docs (e.g. their `skills/docx/SKILL.md`) for guidance on phrasing commands and error handling【51†L739-L748】.  
- **Workflow Orchestration:** Use OpenCode’s ability to invoke subagents (`@agent`) or an orchestrator (e.g. Open Office’s multi-agent) to coordinate these steps. Each subagent would have limited max tokens and tailored functions.  
- **Batch and Summarize:** Implement summarization in the MCP server or as an early agent step, and expose batch tools (OfficeCLI already has `batch` and `merge`). Ensure to **log token usage** at each step (per [64]): track “tokens per tool call” and “context usage per step” to identify bottlenecks【64†L465-L474】.  
- **Benchmarking:** Compare the original one-shot prompt vs. the decomposed pipeline. Key metrics: total tokens per session, time to completion, and output correctness (e.g. validate with `officecli validate`). Test with varied document sizes. Ideally, the multi-agent approach with MCP tools should use a small fraction of the ~130k tokens originally spent, as large file contents are never embedded and most edits happen via JSON commands (zero token cost).  

In summary, by dividing the MD→DOCX task among specialized agents, leveraging OfficeCLI’s MCP capabilities and batch operations, and pruning context aggressively, we can greatly improve performance. Techniques like chain-of-agents【39†L207-L215】 and tree-based task decomposition【43†L53-L62】 guide the design, while OfficeCLI features (template merge, MCP tools) let us avoid token bloat【49†L614-L623】【51†L698-L706】. Monitoring token metrics at each stage will confirm the efficiency gains over the current brute-force approach.
