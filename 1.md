## Chẩn đoán gốc rễ: Tại sao ngốn 130k token?

Vấn đề cốt lõi không phải là officecli command dài. Nó đến từ **3 điểm tích lũy context** đồng thời:

1. **Skill loading toàn bộ**: `officecli-docx/SKILL.md` có size **60 KB** (~15k tokens) — agent load nguyên cái này vào context chỉ để tra cứu một vài command syntax
2. **File content trong context**: Agent đọc nội dung `chuong_2.md` (55 KB) trực tiếp vào context để hiểu rồi fill
3. **O(N²) context growth**: Mỗi lần agent gọi officecli và nhận output, toàn bộ lịch sử tool calls tích lũy trong context. Đây là vấn đề được ghi nhận trong FastAgent framework là *"O(N²) quadratic growth with worker history"*[^1]

Anthropic gọi đây là **"context rot"** — khi số token tăng, model mất khả năng recall thông tin chính xác từ context, dẫn đến performance degradation.[^2]

## Framework lý thuyết để tư duy đúng

Trước khi đề xuất solution, cần hiểu 3 pattern được validate bởi research:

### Pattern 1: Context Folding (arxiv 2510.11967)

Paper *"Scaling Long-Horizon LLM Agent via Context-Folding"*  đề xuất: agent **branch** sang sub-trajectory để xử lý một subtask nặng token, sau đó **fold** nó lại bằng một summary ngắn (~1-2k token). Kết quả: active context nhỏ hơn **10×** trong khi vẫn giữ nguyên performance.[^3][^4]

### Pattern 2: AOrchestra 4-tuple Delegation (arxiv 2602.03786)

Paper  định nghĩa mỗi subagent là tuple **(Instruction, Context, Tools, Model)**. Orchestrator chỉ pass **curated context** cần thiết cho subagent đó — không phải full context. Ablation study (Table 2) cho thấy:[^5]

- **No-Context** subagent → 86% accuracy
- **Full-Context** subagent → 84% accuracy (tệ hơn vì noise)
- **Curated Context** → 96% accuracy[^5]


### Pattern 3: Progressive Disclosure trong Skills

Anthropic skill authoring best practices  chỉ rõ: SKILL.md chỉ load metadata (~50 token) lúc startup. Full content chỉ load khi cần. Nên **split skill thành nhiều file** theo domain, mỗi file chỉ load khi relevant.[^6]

## Giải pháp 1: Refactor Skills theo Progressive Disclosure

**Vấn đề hiện tại**: `officecli-docx/SKILL.md` 60KB (~15k token) load toàn bộ một lúc . Agent không cần biết toàn bộ command encyclopedia ngay từ đầu.

**Giải pháp**: Split `officecli-docx` thành sub-files, SKILL.md chỉ là index:

```
.opencode/skills/officecli-docx/
├── SKILL.md          # ~100 lines: index + quick reference
├── elements.md       # paragraph, heading, table element schemas
├── styles.md         # style/numbering/font commands
├── toc-refs.md       # TOC, references, appendix commands
├── view-query.md     # view, outline, stats, validate commands
└── page-setup.md     # margin, header, footer, page commands
```

**SKILL.md mới** chỉ cần:

```markdown
---
name: officecli-docx
description: Command reference cho OfficeCLI. Load chỉ khi cần syntax
  cụ thể: elements (heading/para/table), styles/numbering, TOC/refs, view/validate.
---
# OfficeCLI Command Reference

## Quick lookup
- **Paragraph/Heading/Table**: [elements.md](elements.md)
- **Style & Numbering**: [styles.md](styles.md)  
- **TOC, References, Appendix**: [toc-refs.md](toc-refs.md)
- **View, Validate, Stats**: [view-query.md](view-query.md)
- **Page, Header, Footer**: [page-setup.md](page-setup.md)
```

Khi agent chỉ cần insert paragraph, nó chỉ đọc `elements.md` (~2k token) thay vì 60KB. **Tiết kiệm ~13k token mỗi lookup**.[^6]

## Giải pháp 2: Tách task thành 3 Subagent với Clean Context

Đây là cải tiến kiến trúc lớn nhất, áp dụng trực tiếp pattern từ AOrchestra và Anthropic.[^7][^2]

**Workflow mới**:

```
User Prompt
    │
    ▼
[ORCHESTRATOR] (Qwen3 35B, ~2k token context)
 - Đọc task.md (~1k token)
 - Nhận biết mode: create / append / rebuild
 - Delegate sang 3 subagent sequentially
    │
    ├──► [SUBAGENT A: md-parser]
    │     Context: chuong_2.md content
    │     Output: structured_content.json (~3-5k token)
    │     → Fold: trả về JSON summary, KHÔNG trả về raw md
    │
    ├──► [SUBAGENT B: template-analyzer]  
    │     Context: format_template.docx outline + styles
    │     Output: style_map.json (numId, ilvl, style names)
    │     → Fold: trả về mapping JSON ~500 token
    │
    └──► [SUBAGENT C: docx-executor]
          Context: structured_content.json + style_map.json (6k total)
          Task: Execute officecli commands theo map
          Output: report.docx
```

**Điểm mấu chốt**: Mỗi subagent nhận **curated context** không quá 10k token. Orchestrator chỉ thấy **kết quả fold** (~2k token), không thấy toàn bộ execution trace của subagent. Đây chính xác là pattern Context Folding từ paper.[^4][^3]

## Giải pháp 3: MCP Server cho Docx — Loại bỏ Token Cost

**Vấn đề**: Mỗi lần agent gọi `officecli` qua shell, output phải chui vào context. 10 lần gọi = 10 tool results tích lũy.

**Giải pháp**: Build một **Python MCP server** dùng `python-docx` hoặc `docxtpl`, expose các tool có tính năng **stateful** (file handle ở server, không ở agent):

```python
# mcp_docx_server.py
from mcp.server.fastmcp import FastMCP
from docxtpl import DocxTemplate
import json, os

mcp = FastMCP("docx-writer")
_open_docs = {}  # server-side state

@mcp.tool()
def open_template(template_path: str, output_path: str) -> str:
    """Mở template và khởi tạo working doc. Trả về handle ID."""
    tpl = DocxTemplate(template_path)
    _open_docs[output_path] = tpl
    return f"OK: handle={output_path}"

@mcp.tool()
def fill_from_json(output_path: str, context_json: str) -> str:
    """Fill toàn bộ document từ JSON context. KHÔNG tính token cho content."""
    tpl = _open_docs[output_path]
    ctx = json.loads(context_json)
    tpl.render(ctx)
    tpl.save(output_path)
    return f"OK: saved {output_path}"

@mcp.tool()
def get_outline(output_path: str) -> str:
    """Lấy outline heading của doc. Token-efficient summary."""
    # ...trả về chỉ heading list, không full content
```

**Tại sao quan trọng**: Server giữ state của document. Agent chỉ truyền **JSON nhỏ** (context variables), không phải toàn bộ nội dung. Kỹ thuật này được MCP-Doc  và word-mcp-server  implement theo hướng này.[^8][^9][^10]

**Cấu hình opencode.json**:

```json
{
  "mcp": {
    "servers": {
      "docx-writer": {
        "command": "python",
        "args": [".opencode/mcp/docx_server.py"]
      }
    }
  }
}
```

## Giải pháp 4: Refactor `docx-from-template` Skill

**Vấn đề hiện tại**: Phase 1 "Analyze" vẫn yêu cầu agent đọc nhiều phần của document để build `analysis` object . Với file 200KB docx, `officecli view outline` có thể dump nhiều token.

**Cải tiến Skill**: Thêm explicit token budget per phase:

```markdown
## Phase Token Budget

| Phase | Max Tool Calls | Max Context Added |
|-------|---------------|-------------------|
| PHASE 0: Preflight | 3 | ~500 token |
| PHASE 1: Analyze | 5 | ~2000 token (outline + styles only) |
| PHASE 2: Execute | 20 | N/A (delegate to subagent) |
| PHASE 3: QA | 8 | ~1000 token |
| PHASE 4: Finalize | 2 | ~200 token |

## Phase 1: Analyze — Forbidden
- KHÔNG được `view content` toàn bộ document
- KHÔNG được `view stats` nếu `view outline` đã đủ
- CHỈ được query style tại điểm neo, không query all styles
```

Anthropic khuyến nghị: **"Tools should return information that is token efficient"** và **"minimal viable set of tools"**. Phase 1 hiện tại vi phạm cả hai nguyên tắc này.[^2]

## Giải pháp 5: Skill Mới `md-to-docx-converter`

Thay vì để agent tự parse markdown và gọi từng officecli command, tạo một **executable script** không cần agent đọc — chỉ cần agent chạy:

```
.opencode/skills/md-to-docx-converter/
├── SKILL.md
└── scripts/
    ├── md_parser.py        # Parse .md → structured JSON
    ├── style_mapper.py     # Analyze template styles → mapping JSON  
    └── docx_builder.py     # Build docx từ JSON + template (python-docx/docxtpl)
```

**SKILL.md**:

```markdown
---
name: md-to-docx-converter
description: Chuyển đổi file .md thành .docx theo template mà không cần
  agent đọc nội dung file. Dùng khi có sẵn template.docx và content.md.
  Token-efficient: agent chỉ cần chạy scripts, không đọc file content.
---

## Workflow (3 bước, không đọc content vào context)

**Bước 1**: Parse markdown thành JSON structure
```bash
python .opencode/skills/md-to-docx-converter/scripts/md_parser.py \
  --input chuong_2.md --output /tmp/content.json
```

Output: /tmp/content.json (agent KHÔNG cần đọc file này)

**Bước 2**: Analyze template styles

```bash
python .opencode/skills/md-to-docx-converter/scripts/style_mapper.py \
  --template format_template.docx --output /tmp/style_map.json
```

**Bước 3**: Build docx

```bash
python .opencode/skills/md-to-docx-converter/scripts/docx_builder.py \
  --content /tmp/content.json \
  --style-map /tmp/style_map.json \
  --template format_template.docx \
  --output report.docx \
  --mode [create|append|rebuild]
```

```

**Tại sao cách này hiệu quả**: Theo Anthropic best practices: *"Utility scripts can be executed via bash without loading their full contents into context. Only the script's output consumes tokens."* Agent chỉ thấy stdout (OK/ERROR), không thấy 55KB content của file.[^6]

## Giải pháp 6: Structured Note-taking trong AGENTS.md

Anthropic khuyến nghị **structured note-taking** cho long-horizon tasks. Cần thêm vào `AGENTS.md`:[^2]

```markdown
## Checkpoint State Management

Agent PHẢI ghi checkpoint vào `/tmp/docx_state.json` sau mỗi phase:
```json
{
  "session_id": "...",
  "phase_completed": 2,
  "working_file": "report.docx",
  "style_map_cached": "/tmp/style_map.json",
  "content_parsed": "/tmp/content.json",
  "issues": []
}
```

Nếu session bị ngắt, agent đọc checkpoint và resume từ phase cuối.
KHÔNG re-analyze template nếu style_map_cached đã tồn tại.

```

Đây chính xác là **structured note-taking pattern** từ paper Anthropic  — agent dùng file ngoài context để giữ state, không phải re-load vào context.[^2]

## Tổng hợp: Kiến trúc đề xuất

```

User: "Tạo report.docx từ chuong_2.md theo format_template.docx"
│
▼
AGENTS.md (routing) ~3k token
│
▼
skill: md-to-docx-converter (SKILL.md ~300 token)
│
┌─────────┼─────────┐
▼         ▼         ▼
Script A  Script B  Script C
(parse)  (analyze) (build)
No context  No context  No context
cost        cost        cost
│         │         │
└────► /tmp/*.json ◄─┘
│
▼
report.docx ✓

Total LLM context: ~8-12k token (vs 130k hiện tại)

```

**So sánh trước/sau**:

| Metric | Hiện tại | Đề xuất |
|--------|----------|---------|
| Context per session | ~130k token | ~8-15k token |
| officecli-docx load | 60KB mỗi lần | Chỉ sub-file relevant (~3-8KB) |
| MD content in context | 55KB trực tiếp | 0 (script xử lý) |
| Tool call accumulation | O(N²) | O(1) per subagent |
| Resume on crash | Không | Có (checkpoint JSON) |
| Multi-case support | Cần re-prompt | Mode flag: create/append/rebuild |

1. **Ưu tiên cao**: Split `officecli-docx/SKILL.md` thành sub-files — effort thấp, impact ngay lập tức (~13k token saved mỗi session)[^6]
2. **Ưu tiên cao**: Viết `md_parser.py` + `docx_builder.py` dùng `python-docx` hoặc `docxtpl`  — đây là giải pháp loại bỏ **55KB content** ra khỏi context[^11][^12]
3. **Ưu tiên trung bình**: Thêm token budget constraints vào `docx-from-template/SKILL.md` 
4. **Ưu tiên trung bình**: MCP server cho stateful docx operations[^10][^8]
5. **Ưu tiên thấp**: Orchestrator-subagent decomposition nếu cần scale thêm[^5]

Với chỉ bước 1 + 2, ước tính context sẽ giảm từ 130k xuống **~20-30k token** — đủ để Qwen3 35B hoạt động ổn định trong context window của nó mà không bị context rot.[^3][^2]
